# Agent Orchestration

## Overview

병렬 작업 시 subagent와 agent team 중 목적에 맞는 아키텍처를 선택하고, 팀을 낭비 없이 운영하기 위한 판단 기준.

## 아키텍처 선택

| 상황 | 선택 |
| --- | --- |
| 테스트 실행, 문서 가져오기, 로그 파일 처리 등 빠르고 집중된 워커 | subagent |
| 팀원이 결과를 공유하고, 서로 도전하고, 독립적으로 조정해야 함 | agent team |
| 병렬 subagent가 컨텍스트 제한에 도달하거나 서로 통신 필요 | agent team으로 전환 (터닝포인트) |

- 작업이 병렬 작업의 이점이 있다고 판단하면 팀 생성을 제안할 것
- agent team 구성 전 내장 인터뷰 도구로 사용자에게 질문하여 프로젝트에 최적화되도록 구성

## Agent Team 운영 수칙

- 리더는 작업을 나누고 팀원에게 할당할 때 작업을 적절히 크기 조정
- 병렬 코드 검토: 검토 기준을 독립적인 도메인으로 분할, 각 팀원에게 겹치지 않는 고유한 렌즈 할당
- 경쟁하는 가설로 조사: 팀원들을 명시적으로 적대적으로 구성 (토론 구조가 핵심 메커니즘) — 독립 조사자가 서로의 이론을 적극 반박할 때, 생존하는 이론이 실제 근본 원인일 가능성이 훨씬 높음
- 팀원에게 충분한 컨텍스트 제공, 완료될 때까지 대기
- 파일 충돌 방지: 각 팀원이 다른 파일 집합을 소유하도록 작업 분할
- 모니터링 및 조율: 진행 상황 확인, 작동하지 않는 접근 재지정, 발견 종합 — 무인 방치가 길수록 낭비 위험 증가

## 팀 크기

- 대부분의 워크플로우는 3-5명으로 시작 (병렬 작업과 관리 가능한 조율의 균형)
- 팀원당 작업 5-6개 유지 — 독립 작업 15개면 팀원 3명이 좋은 시작점
- 동시 작업의 실질적 이점이 있을 때만 확장

## 비용

- 팀원 모델은 sonnet 기본 (기능·비용 균형), 간단한 subagent는 더 저렴한 모델 고려
- spawn 프롬프트는 집중적으로 유지
- 작업이 완료되면 팀을 정리
